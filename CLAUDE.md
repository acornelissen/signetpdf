# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project status

SignetPDF is a minimal, free, cross-platform PDF viewer for filling forms, editing text and signing. The walking skeleton (M0) is under way: the Tauri 2 + Vite + TypeScript app builds and runs, but feature work (PDF rendering, forms, annotations, signing) is still ahead. The work plan is tracked in beads (see Task tracking below). The architecture below is the planned target; build it as the plan describes rather than inventing a different structure.

## Task tracking (read before starting work)

The work plan lives in **beads**. The system of record is the local Dolt database under `.beads/`; [.beads/issues.jsonl](.beads/issues.jsonl) is the passive, git-tracked export (regenerated by `bd export`, not hand-edited). Use the `bd` CLI to read and update issues — never edit the JSONL directly. See the Beads Issue Tracker section below for the command reference.

- Work is organized into milestones M0–M5, each an `epic` with child tasks and a **gate** task (`*-gate`, label `gate`) that defines its Definition of Done.
- Tasks carry hard `dependencies`. A milestone is blocked until the prior milestone's gate closes. Respect this ordering — do not start M2 work while M1's gate is open.
- Most tasks specify "Test first" and a conventional-commit message. Honor both: the commit subject in the task is the intended commit.

## Planned toolchain and commands

- brew and mise for package management
- **Shell:** Tauri 2 (Rust backend in `/src-tauri`).
- **Frontend:** Vite + **TypeScript**. No UI framework — the DOM is driven directly. Model and seam types are real TS interfaces/types, checked by `tsc`.
- **PDF read/render:** PDF.js (`pdfjs-dist`). **PDF write:** pdf-lib. These are two different libraries with two different jobs; keep them on their respective sides of the seam (see below).
- **Tests:** Vitest, headless. **Types:** `tsc --noEmit` in CI. **Lint:** ESLint + Prettier. **License:** Apache-2.0 with the Commons Clause (non-commercial; the software may not be sold).

Once scaffolded (M0), expect:

```
npm run tauri dev     # launch the desktop app
npm test              # Vitest suite (headless)
npm run typecheck     # tsc --noEmit
npm run lint          # ESLint + Prettier
```

Run a single test with Vitest's filters, e.g. `npx vitest run path/to/file.test.ts` or `npx vitest run -t "test name"`. Rust unit tests (for the Tauri commands) run with `cargo test` inside `/src-tauri`.

CI is a GitHub Actions matrix across **ubuntu/macos/windows**; every gate requires the matrix green on all three.

## Architecture: the three invariants

The entire design hangs on three rules. Most bugs in this codebase will be violations of one of them.

### 1. One immutable DocumentModel is the single source of truth

```ts
// All geometry is PDF user space (points, bottom-left origin) — NEVER screen pixels.
interface DocumentModel { sourceBytes: Uint8Array; fieldValues: FieldValue[]; annotations: Annotation[]; pages: PageGeometry[]; dirty: boolean }
interface FieldValue { kind: 'field'; fieldName: string; value: string | boolean }
interface TextBox { kind: 'text'; id: string; page: number; origin: Point; width: number; height: number; text: string; fontSize: number }
interface SignatureStamp { kind: 'signature'; id: string; page: number; origin: Point; width: number; height: number; pngBytes: Uint8Array }
type Annotation = TextBox | SignatureStamp   // discriminated on `kind`
interface PageGeometry { index: number; width: number; height: number; rotation: number }   // user-space units, from PDF.js /Rotate
interface Point { x: number; y: number }
```

The model is `readonly` end to end (use `Readonly`/`ReadonlyArray` or `as const` factories) so the compiler enforces invariant 1 below — a mutation that tries to write a field is a type error, not just a convention.

Every mutation — `setFieldValue`, `addAnnotation`, `updateAnnotation`, `removeAnnotation` — returns a **new** model, never touches the input, and sets `dirty=true`. IDs come from one central helper. Form-field controls and annotation widgets hold **no state of their own**: every change routes through a model mutator and the UI re-renders from the model. Undo/redo (M5) is just a stack of these immutable snapshots.

### 2. The coordinate seam lives in exactly one module

`modelToScreen(point, pageGeometry, viewport)` converts user-space (bottom-left origin) to screen pixels (top-left origin), honoring scale and rotation (0/90/180/270). `screenToModel` is its exact inverse, in the **same module**. This is the project's highest-risk math; it is DRY by mandate. The inverse is verified by a property test asserting `screenToModel(modelToScreen(p)) ≈ p` across all rotations and scales.

Any time the UI turns a click into a position, or draws an overlay over a field, or places a stamp, it goes through this seam. **Rendering code must contain no annotation or field state** (an explicit M1 gate condition) — it reads the model and the seam, nothing more.

### 3. Save is a pure projection: model → pdf-lib → bytes

The save path loads `sourceBytes` with pdf-lib and applies the model: field values via `getForm()`, text via `drawText` (with an embedded Unicode-capable font — pdf-lib's standard fonts are Latin-only), signatures via `embedPng`+`drawImage`. It is **pure and DOM-free** so it can be unit-tested. Correctness is proven by **golden-file round-trip tests**: save, re-open, assert the content survived. The baseline is that an empty model round-trips to an identical page count and AcroForm field set.

## Other rules baked into the plan

- **XFA is refused, not rendered.** Detect XFA forms and show a clear message. Only AcroForm is supported.
- **PDF.js worker must be correctly bundled** so its path resolves inside the Tauri webview. This is an M0 task flagged "sort now, not later" — a broken worker path is the classic Tauri+PDF.js failure.
- **Rust owns file I/O.** Tauri commands `open_pdf`, `save_pdf`, `save_pdf_as` do native dialogs and byte read/write; each has a Rust unit test.
- **Fixtures matter.** `/fixtures` holds a deliberately awkward corpus (linearized PDF, a `/Rotate 90` page, an AcroForm with every field type, a no-forms PDF, an XFA PDF for refusal tests), documented in `fixtures/README`. Tests lean on these.
- **Binary code-signing/notarization is out of scope for 0.1.0** and documented as a known gap (macOS unnotarized, Windows unsigned).

## Agent Context Profiles

The managed Beads block is task-tracking guidance, not permission to override repository, user, or orchestrator instructions.

- **Conservative (default)**: Use `bd` for task tracking. Do not run git commits, git pushes, or Dolt remote sync unless explicitly asked. At handoff, report changed files, validation, and suggested next commands.
- **Minimal**: Keep tool instruction files as pointers to `bd prime`; use the same conservative git policy unless active instructions say otherwise.
- **Team-maintainer**: Only when the repository explicitly opts in, agents may close beads, run quality gates, commit, and push as part of session close. A current "do not commit" or "do not push" instruction still wins.

## Session Completion

This protocol applies when ending a Beads implementation workflow. It is subordinate to explicit user, repository, and orchestrator instructions.

1. **File issues for remaining work** - Create beads for anything that needs follow-up
2. **Run quality gates** (if code changed) - Tests, linters, builds
3. **Update issue status** - Close finished work, update in-progress items
4. **Handle git/sync by active profile**:
   ```bash
   # Conservative/minimal/default: report status and proposed commands; wait for approval.
   git status

   # Team-maintainer opt-in only, unless current instructions forbid it:
   git pull --rebase
   git push
   git status
   ```
5. **Hand off** - Summarize changes, validation, issue status, and any blocked sync/commit/push step

**Critical rules:**
- Explicit user or orchestrator instructions override this Beads block.
- Do not commit or push without clear authority from the active profile or the current user request.
- If a required sync or push is blocked, stop and report the exact command and error.
